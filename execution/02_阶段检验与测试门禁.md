# 02_阶段检验与测试门禁

## 目标
- 明确每个M阶段的测试策略、关键测项、通过门槛与失败应对，确保各阶段交付具备可追踪的质量保障。
- 在功能、接口、性能、安全、可用性五个层级上逐步扩充验证深度，配套回归要求与验收签字流程。
- 以门禁逻辑推进，从低风险的烟雾验证到面向客户的交付验收，形成清晰的质量传递链条。

## 分层说明
1. 功能：核心业务流程、边界条件和数据流完整性。
2. 接口：API/合约与内部集成点的正确性与版本兼容。
3. 性能：响应/吞吐、容量与基本稳定性指标。
4. 安全：代码/配置静态扫描、权限校验、依赖清单。
5. 可用性：启动、部署、监控报警、恢复路径。

---

## M1 – 构建与基础烟雾验证
### 测试策略
- 确保最新构建成功启动并能支撑最小业务路径。
- 快速验证核心功能、接口和启动链路，以阻断明显回归。

### 测试矩阵
| 层级 | 关键测项 | 方法 | 通过阈值 |
| --- | --- | --- | --- |
| 功能 | 核心业务流程 (如采集-调度-反馈) | 单个工作流冒烟、自动化脚本 | 不出现致命异常，流程能走完关键节点 |
| 接口 | 主API/消息通道契约 | Schema校验、mock端点 | 请求成功率100%、无schema破坏性变更 |
| 性能 | 单请求响应 | Cloudwatch/本地profiling | 单个请求响应<2s，无崩溃 |
| 安全 | 基本静态扫描、依赖白名单 | SAST/依赖扫描 | 不存在政策级别CVSS>=7漏洞 |
| 可用性 | 服务启动与依赖连通 | 环境自检 | 依赖服务 ping/health check 均通过 |

### 通过阈值
- 所有核心功能冒烟通路完成、接口无schema破裂、主要服务稳定启动。

### 失败处理
- 失败即回滚至上一稳定构建；记录在Smoke Bug List，并要求开发立即修复后重新触发M1验证。
- 若属于依赖层面（如数据库启动失败），由环境团队先恢复基础链路。

### 回归要求
- Smoke失效后，需重新运行M1自动化集中集作为回归验证；在M1与M2间不能放行未重新跑过的冒烟点。

### 验收签字流程
- 研发负责人确认构建健康 + QA负责人声明Smoke完成 + 技术运营签名。

## M2 – 模块集成与关键能力确认
### 测试策略
- 在M1基础上增加完整业务模块的交互/接口连通性验证，覆盖更丰富数据场景。
- 引入边界异常、授权校验等非Happy Path测试。

### 测试矩阵
| 层级 | 关键测项 | 方法 | 通过阈值 |
| --- | --- | --- | --- |
| 功能 | 关键子系统交互 (调度、统计、告警) | 集成测试 + 端到端验证 | 子系统间调用成功率≥98%，无致命缺陷 |
| 接口 | 内部服务间契约 + Gateway | Contract tests + API探测 | 版本兼容，无接口失效 |
| 性能 | 关键接口压力预判（峰值1/4） | 压测脚本 | 平均响应 < 2.5s，error rate<1% |
| 安全 | 身份/权限验证、输入校验 | 自动化安全扫描 + 手工点验 | 无高危漏洞、权限漏斗闭合 |
| 可用性 | 集成部署过程与健康 | 部署跑批 + 灾备切换演练 | 切换路径成功，指标有监控覆盖 |

### 通过阈值
- 集成路径通过，接口契约未破坏，性能与健康检查在预设抖动范围内。

### 失败处理
- 归类为功能/接口/性能/安全/可用性，由对应团队指定责任人，限制未解决问题不得推进。
- 失效项需形成补丁 + 回归证明（M1、M2全量重新跑前需评估风险）。

### 回归要求
- 修复后必须重跑M2全套测试；影响上下游的模块还需并行执行M1用例验证底层稳定性。

### 验收签字流程
- 模块owner、QA lead、产品代表签字确认。

---

## M3 – 场景测试与性能准入
### 测试策略
- 用接近真实业务的数据与流量，对主要能力做场景级验证。
- 增强性能、容量、可用性压力验证，确保在预期峰值下系统不崩溃。

### 测试矩阵
| 层级 | 关键测项 | 方法 | 通过阈值 |
| --- | --- | --- | --- |
| 功能 | 多任务并发、异常恢复 | 场景脚本 + Chaos | 所有主要场景通过，恢复时间≤60s |
| 接口 | 外部对接 (第三方/客户端) | 接口回归 + 端到端链路 | 100%请求成功率，超时率<0.5% |
| 性能 | 峰值3/4压力 + 容错 | 性能测试平台（JMeter/Locust） | 吞吐达标、核心指标稳定、无内存泄漏 |
| 安全 | 动态扫描 + 身份漏斗 | DAST + 权限漏测脚本 | 不存在认证绕过、敏感数据泄露 |
| 可用性 | 自动化部署、切换、指标 | 蓝绿/金丝雀 + 报警漏斗 | 部署零中断，报警自动触发且联动通知 |

### 通过阈值
- 场景脚本全部通过，性能/可用性指标在5分钟抖动范围内；安全审批已过。

### 失败处理
- 阶段内发现拒收级缺陷需暂停所有后续M3验证，待修复后择机重跑对应场景；对于性能波动需作Root Cause Report。

### 回归要求
- 修复后必须跑完M3场景集和关键性能载荷；若性能触发，需回退验证性能/detect并补充监控告警。

### 验收签字流程
- 技术负责人、SRE、QA经理签字；性能或安全负责人按需补签。

## M4 – 客户可用度与安全实战演练
### 测试策略
- 视野拉高至客户可见能力，执行接口稳定性、用户体验、权限控制与安全演练。
- 包括回归验证、灾备验证、监控告警完整性测试。

### 测试矩阵
| 层级 | 关键测项 | 方法 | 通过阈值 |
| --- | --- | --- | --- |
| 功能 | 客户主要旅程 + 兼容性 | 手动 + 自动化复盘 | 全流程成功率≥99%，关键路径无缺陷 |
| 接口 | 外部依赖 / 数据同步 | 实际对接验收 | 同步延迟≤1.5倍SLA，失败自动重试 |
| 性能 | 客户场景负载 + 峰值验证 | 实流量回放 | 核心指标符合SLA，调用链不超时 |
| 安全 | 渗透 + 数据合规 | 红队场景 + SOC check | 无安全事件、日志可追溯 |
| 可用性 | 灾备 / 回滚流程 | 实操演习 | 可用率达SLA，无全链路断点 |

### 通过阈值
- 客户旅程无缺陷，SLA指标满足，监控与告警触发可靠。

### 失败处理
- 任一层级验收失败即刻拉起高优先级事件，由SRE与相关owner联合通报；必要时切换到备用方案或回退。

### 回归要求
- 所有修复后必须按M4脚本复测，涉及安全/合规需附上复测截图与记录。

### 验收签字流程
- 产品经理、客户成功、运营负责人确认 + 安全负责人验收。

---

## M5 – 正式交付与持续运营
### 测试策略
- 聚焦于可运营性、长期稳定、扩展性 & 监管要求，形成完整的交付验收。
- 强调回归自动化覆盖、性能/安全门控、监控/报警信号。

### 测试矩阵
| 层级 | 关键测项 | 方法 | 通过阈值 |
| --- | --- | --- | --- |
| 功能 | 全量业务流程 + 扩展场景 | 全量回归 + 变更覆盖 | 回归覆盖率≥95%，无高优先级遗留 |
| 接口 | 公共API/版本升级计划 | 版本兼容 + 公告验证 | 客户接口兼容性无断裂 |
| 性能 | 24h稳定性 + 7x24监控 | 长期观测 + 压测复盘 | 平均负载在目标内，资源利用率稳定 |
| 安全 | 合规审计 + SOC | 外部审计 + 自动化策略 | 无新高危漏洞，审计通过 |
| 可用性 | 监控/告警/灾备全链 | 运行演练 + 指标实测 | 报警响应时间可控，SRE runbook可用 |

### 通过阈值
- 交付文档、监控链路、告警响应、回归自动化全部落地；运营团队接手时指标与录制演练均可再现。

### 失败处理
- 重大问题需启动发布回滚策略；中等级别问题必须在48小时内修补并验证，必要时延期交付。

### 回归要求
- 修复后必须在M5全量环境中完成回归并同步到自动化发布流水线；新增监控或告警需同步联调。

### 验收签字流程
- 交付负责人、运营/客户成功、QA主管、合规/安全代表签字，形成M5验收手册记录。
